## What Went Well ✅

### 1. The mediamtx Discovery and Pivot

**What happened:** Hardware QA of Story 3.2 revealed two structural defects:
- `rpicam-vid --listen` coupled camera hardware lifecycle to network client connections — any server restart caused a libcamera SIGABRT and camera device lock
- HLS structural latency was 10–15 seconds regardless of transport optimizations

**The pivot:** Replaced `rpicam-vid + raw TCP + ffmpeg + HLS` with `mediamtx` on both sides:
- Pi side: mediamtx rpiCamera source → RTSP at `:8554` (camera always on, clients connect/disconnect freely)
- Server side: mediamtx pulls RTSP via frp → republishes as WebRTC WHEP → sub-second browser latency

**Why it matters:** We didn't try to fix HLS. We recognized a structural limitation and replaced it cleanly. The result is objectively better than the original plan: a home camera should be *live*, not a 15-second tape delay.

**Process win:** The formal 3-2b and 3-2c story documents made the pivot fully traceable. The retrospective notes embedded in those files (`"The --listen anti-pattern: any tool that couples a hardware resource lifecycle to a network client connection is fragile by design"`) are portable architectural wisdom.

---

### 2. Hardware QA as an Explicit Gate

Both mediamtx pivots were caught during hardware QA of Story 3.2 — *before* Stories 3.3–3.6 built the SPA on top of a broken foundation.

If the defects had slipped through, Stories 3.3 (stream player), 3.4 (WS hub), and 3.5 (admin toggle) would all have been built on HLS. Ripping that out mid-epic would have been significantly more disruptive.

**Lesson:** Hardware QA is a first-class gate, not an optional step.

---

### 3. WebSocket Infrastructure Composing Cleanly

The WS hub pattern from Story 3.4 composed seamlessly with Story 3.5 admin toggle:
- `streamService.setAdminToggle()` calls `broadcastState()` calls `wsHub.broadcast()`
- Every connected client updates instantly — no additional wiring required

The `useWebSocket.ts` composable's `onmessage` handler (type-check → dispatch) is already extensible for Epic 4 `chat:message` types without any refactoring.

## Challenges & Learnings ⚠️

### 1. UX Spec Gap — Interaction Flows vs. Visual Mockups

**The issue (Story 3.6):** The UX design spec and story ACs were approved pointing toward a certain desktop interaction model. The implementation faithfully followed the spec. Code review identified a fundamental UX problem — "Camera Controls" in the profile menu on both desktop and mobile was a "feature mishmash" that didn't align with the desktop hover-overlay UX. The fix (adding a dedicated `|→` / `←|` toggle button in the stream overlay) was correct but touched multiple components: `StreamPlayer.vue`, `WatchView.vue`, `AdminPanel.vue`, and `ProfileAnchor`.

**Root cause:** The UX spec documented *visual states* (color tokens, component names, layout measurements) but not *interaction flows* (step-by-step: admin opens panel → does X → panel appears → closes → does Y). For hover-gated desktop UIs, the transitions are harder to specify than the static states.

**Lesson:** Any story with hover/open/close state transitions needs an explicit interaction flow written in the story before dev starts — not just mockup screenshots.

---

### 2. Spec Document Drift — Wrong Package Paths and Obsolete ACs

Two stories had incorrect technical details in their spec documents:
- **Story 3.4:** Specified `@hono/node-server/ws` as the import path. This package subpath doesn't exist. Correct package is `@hono/node-ws`. Required investigation before implementation could begin.
- **Story 3.3:** Epics.md ACs referenced `hls.js` and `/hls/stream.m3u8` — both obsolete after 3-2c. The story itself had a correction note, but the underlying epics.md remained stale.

**Lesson:** When spec errors are discovered during dev, correct them *in the source document* immediately — not just worked around silently. Story files and epics.md should reflect ground truth at all times.

---

### 3. Architecture Document Assumptions Not Hardware-Validated

The original architecture assumed `rpicam-vid --listen` provided a stable TCP server interface. This assumption was never validated against actual hardware behavior before Story 3.1 was implemented. The defect was discovered one story later.

**Lesson:** For hardware-dependent architecture decisions, flag the assumption explicitly in the architecture doc and identify the earliest story where hardware validation is possible. If the assumption is wrong, the earlier we know, the less we've built on top of it.

---

### 4. Tech Debt from mediamtx Pivot

The pivot to mediamtx was correct but left orphaned artifacts:
- `hls.js` still in `apps/web/package.json` (never used post-pivot)
- Pi agent camera pipeline loop has no restart delay — libcamera needs 2–3s to release device after mediamtx crash
- mediamtx not yet added to server Dockerfile (Story 6.x)
- WebRTC UDP ICE for production NAT not configured (Story 6.x)

These are known and logged — not surprises, but they need clearing before or during Epic 6.

## Significant Discovery: Go Agent Scope Collapse ⚠️

**Discovery surfaced during retrospective:**

The original Pi agent was designed as:
1. An HTTP API server for `v4l2-ctl` camera parameter changes (primary purpose)
2. A supervisor for frpc tunnels (bonus QOL)

After the mediamtx pivot:
1. mediamtx owns the camera pipeline natively — no Go API needed
2. Story 3.6 camera controls wire directly to mediamtx's HTTP API (`127.0.0.1:9997`) — the Go agent HTTP server was never built
3. The agent now only launches mediamtx and frpc as supervised subprocesses — both of which could be systemd services with static configs

**The agent is now a thin subprocess launcher for things that could be systemd units.**

Additionally, the Epic 6 captive portal story (6-3) can be replaced entirely by [wifi-connect](https://github.com/balena-os/wifi-connect) — a battle-tested, balena-maintained project that handles WPA supplicant complexity that would have been significant custom Go implementation.

**Decision:** Leave the agent as-is. Transform Epic 6 from "implement agent features" to "get this documented extremely well with robust install/uninstall scripts."

2. **Interaction flows must be specified, not just visual mockups.** The 3-6 UX rework was caused by a spec gap in desktop open/close transitions — not a dev error. Static mockups show states; flows show transitions.

3. **The mediamtx pivot was a net positive.** Recognizing a structural limitation and replacing it cleanly — rather than patching around it — is engineering discipline.

4. **Epic 6 needs a planning reset.** The agent's reduced scope after mediamtx changes what Epic 6 should deliver. Catch this now rather than mid-epic.

5. **Spec errors belong in the spec, not in workaround notes.** When the wrong package path or obsolete AC is discovered, fix the source document.
