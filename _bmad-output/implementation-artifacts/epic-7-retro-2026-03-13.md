## Challenges & Learnings ⚠️

### 1. Story 7-4 Scope Was Too Large

**What happened:** Story 7-4 had 14 acceptance criteria covering server TCP infrastructure, WebSocket broadcasting, client composable, AND a 5-state UI component.
**Lesson:** Stories with >10 ACs should be proposed for split during sprint planning. Large stories create testing blind spots and review fatigue.
**Impact:** 2 hotfixes required post-ship (FRP host hardcoded, smart-charge logic bug).

### 2. Hardware Verification Gaps

**What happened:** 7-4 shipped without actual PiSugar hardware verification. Tests mocked TCP connections but didn't simulate Docker network topology.
**Lesson:** Hardware-dependent stories need explicit verification gates. Mocks pass but don't catch environment-specific bugs.
**Impact:** FRP host bug only manifested in Docker where PiSugar is on FRP host, not localhost.

### 3. Inconsistent Code Review Documentation

**What happened:** Only 7-1 has a formal code review findings document. Other stories had inline review notes but no separate document.
**Lesson:** Large stories (>8 ACs) should require formal findings documentation for traceability.
**Impact:** Harder to audit review thoroughness retroactively.

### 4. BroadcastConsole Styling Deviation

**What happened:** 7-1 code review noted BroadcastConsole uses theme background instead of specified `bg-black/40 backdrop-blur-sm`.
**Lesson:** Document intentional deviations during implementation, not in retro.
**Impact:** Minor aesthetic inconsistency, documented as intentional.
