### 1. `roleUtils.ts` Composed Exactly As Designed

Story 5-1 explicitly noted: "Do NOT inline this in `chatService.ts` — it will be reused in Stories 5-2 and 5-3." `canModerateOver()` and `ROLE_RANK` were imported cleanly in mute (5-2), ban (5-3), and gating (5-4) without any refactoring. The forward-thinking placement paid off immediately across four stories.

### 2. Story 5-1 — First Fully Clean Code Review in the Project

Zero critical, high, medium, or low findings. The careful architecture choices — two distinct 403 codes (`FORBIDDEN` vs `INSUFFICIENT_ROLE`), client-side and server-side gating in parallel, AlertDialog pattern, ContextMenu mobile compatibility — all held up under adversarial scrutiny. A genuine milestone.

### 3. `computeUserTag` Centralization Fixed a Real Functional Bug

Story 5-6 identified and fixed an actual production bug: `computeUserTag` existed as three separate private functions across `chatService.ts`, `ws.ts`, and `userService.ts` — and **none of them implemented the `ViewerGuest` default tag**. Centralizing into `apps/server/src/lib/user-tag.ts` fixed the bug and eliminated the duplication. The bug would have been invisible in production until a ViewerGuest sent a chat message.

### 4. Ban Architecture Was Solid End-to-End

Story 5-3's `wsHub.revokeUserSessions()` + `prisma.$transaction()` pattern is exactly the right design: atomic DB operation (update bannedAt + delete sessions), immediate WS connection closure, `session:revoked` message type causing instant client redirect. The architecture held together cleanly.

### 5. +211 Tests With Zero Regressions

527 → 738 tests across the epic. No regressions despite touching auth middleware, WS hub internals, CLI tooling, a new color picker component, admin API endpoints, and three composables. The testing floor established in Epic 4 continued to hold.

## Challenges & Learnings ⚠️

### 1. Sequencing Recommendation Ignored — Got Lucky

**What happened:** The Epic 4 retro explicitly flagged Story 5-5 (four-tier role hierarchy, CLI, web UI) as the highest-risk story and recommended sequencing it early — "everything permission-gated is downstream of it." In practice, 5-5 was implemented fifth.

**Why it didn't block us:** The existing `Role` enum from Epic 1/2 was sufficient for 5-1 through 5-4. `hasRole()` from `packages/types` wasn't required by any of them. This was fortunate, not planned.

**Root cause:** Retro recommendations are recorded but not enforced in sprint planning. If the recommendation appears in the retro, it must appear as a constraint in the next sprint plan — not as a note to consider.

**Lesson:** Sprint planning must explicitly reflect retro sequencing flags. "Sequence early" in a retro = first story in the next epic, not a suggestion.

---

### 2. Spec Translation Gap — Intent Lost Between Vision and AC

**What happened (5-6):** The full Reka UI color picker (ColorArea gradient + hue slider + ColorField hex input + swatch picker) was the original design intent from Zikeji. The story AC captured only "12 color swatches." The agent implemented swatches first (correctly matching the AC), then had to pivot to the full picker mid-story.

**Root cause:** `create-story` only captured a simplified proxy of the actual UI intent. The visual/interaction design intent was not surfaced — likely because `ux-design-directions.html` (which contains visual mockups) is not auto-loaded by the `discover_inputs` protocol (it only globs `*.md` files).

**Impact:** Extra implementation round, spec divergence documented as "approved pivot," agent had to recover mid-story rather than building it right the first time.

**Lesson:** For any story with specific UI component design intent, the story author must explicitly reference `ux-design-directions.html` and capture the actual interaction model in Dev Notes — not a simplified AC proxy. This gap was noted in the BMAD Workflow Notes from a prior epic and still caused a problem here.

---

### 3. Agent Comprehension Failure — Basic UX Semantics

**What happened (5-6):** The "Clear" button on the UserTag popover required significant back-and-forth. The expected behavior is simple: clicking Clear resets the local form fields (input text, selected color) immediately, then calls the API to clear the server value. The agent implemented the API call but failed to reset the local UI state — requiring a correction round.

**Why this is notable:** This is not an architectural failure. The agent handled `prisma.$transaction`, WS broadcast patterns, and Reka UI component integration correctly. It missed a piece of intuitive UX behavior that any human would understand without needing a spec.

**Root cause:** Agents handle structural complexity better than they model user-facing behavioral expectations. "Clear" on a form is an implicit contract that doesn't appear in the AC.

**Lesson:** For form-like UIs with Clear/Reset behavior, the story Dev Notes must explicitly state what "Clear" means for local UI state: *"Clear: reset local form fields to empty immediately (do not wait for API response), then call API."* Do not assume the agent infers standard UX conventions.

---

### 4. Story 5-2 HIGH Finding — Missing `credentials: 'include'`

**What happened:** Mute/unmute API calls were made without explicit `credentials: 'include'`, meaning the session cookie would be silently dropped in cross-origin contexts. Caught in code review, fixed — but this class of bug passes all unit tests (tests mock `apiFetch` entirely) and would only manifest in a deployed environment.

**Lesson:** WS-effect stories involving new `apiFetch` calls should have a dev note explicitly flagging: *"Ensure `credentials: 'include'` is passed — all `apiFetch` calls must include session cookie."* The existing wrapper handles this, but any story adding new fetch calls should verify.

---

### 5. Story File Status Not Updated at Closure

**What happened:** Story 5-5 file header reads `Status: ready-for-review` but `sprint-status.yaml` records it as `done`. The sprint-status is authoritative, but the story file is the human-readable record.

**Lesson:** The dev agent must update the `Status:` field in the story file to `done` before closure. This is a simple discipline check that should be part of the story completion checklist.

### Key Takeaways

1. **Shared lib patterns work.** `roleUtils.ts` composing across four stories without rework is the right model. Every cross-story utility belongs in a named lib, not inlined.

2. **Agents handle architecture well; basic UX semantics require explicit spec.** The Clear button failure is a reminder: do not assume agents infer standard form behavior. State it explicitly in Dev Notes.

3. **Spec translation is a create-story discipline, not a dev-story problem.** Intent lost between "full color picker" and "12 swatches" happens at story creation time. The fix is in `create-story`, not in code review.

4. **Retro sequencing flags are constraints, not suggestions.** If a retro says sequence X early, it goes in the sprint plan as story #1 — not as a note to consider.
