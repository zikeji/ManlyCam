## Challenges & Learnings ⚠️

### 1. **Scaffold Tools Require Manual Verification (Story 1.1)**

**The Issue:**
- `pnpm create vite@latest` created vanilla TypeScript instead of `vue-ts` template → had to reconfigure manually
- `@vitejs/plugin-vue` version conflict with Vite 7 (peer dep warning)
- `shadcn-vue init` is interactive; had to create `components.json`, `src/assets/index.css`, `src/lib/utils.ts` manually

**The Lesson:**
Automated scaffolds don't always match expectations. Manual verification checklist before proceeding helps:
- [ ] Verify scaffold language matches requirement (TypeScript vs. vanilla JS)
- [ ] Check peer dependency warnings in `npm list` / `pnpm list`
- [ ] Run `pnpm install`, `pnpm build`, `pnpm typecheck` before committing

**Impact:** Minimal — caught during implementation, not code review.

---

### 2. **Database Indexes Aren't Automatic (Story 1.2)**

**The Issue:**
Initial Prisma schema lacked `@@index` annotations on foreign key columns. This would cause table scans on queries like:
```sql
SELECT * FROM sessions WHERE user_id = ?  -- without index, full table scan
```

**The Lesson:**
Prisma doesn't auto-index FKs. Best practice:
- Every `@relation` with `fields: [fkField]` should have a corresponding `@@index([fkField])`
- Code review should explicitly check FK columns for indexes
- Performance testing early catches these issues

**Impact:** Caught in code review, fixed immediately. Would have been critical in production.

---

### 3. **Architecture Specs Need Cross-Validation (Story 1.4)**

**The Issue:**
Story 1.1 created `config.example.toml` with field names (`fps`, `bitrate`, `device`) that didn't match `architecture.md` spec (`framerate`, `codec`, `output_port`). Story 3.1 would implement Go struct against architecture — two sources of truth.

**The Lesson:**
When multiple documents define the same structure, pick ONE authoritative source and reference it everywhere:
1. Architecture doc is authoritative (defines what the system *should* do)
2. Implementation docs reference and reproduce the spec
3. Code review explicitly validates alignment

**Impact:** Resolved in Story 1.4 code review. Risk averted for Story 3.1 (agent implementation).

---

### 4. **Tailwind v3 Pin Was Necessary (Story 1.1)**

**The Insight:**
`shadcn-vue` doesn't have stable v4 support yet. The decision to pin `tailwindcss@3` was correct and saved integration headaches.

**The Lesson:**
Component library compatibility matters. Check before upgrading dependencies.

   - Prerequisites: Google OAuth profile data (Story 2.1)

5. **Story 2.5:** CLI allowlist + ban management
   - `manlycam-admin allowlist add-domain company.com`
   - `manlycam-admin users ban user@example.com` (atomically revokes sessions)
   - Prerequisites: CLI framework (Story 1.1 ✅), transaction patterns
