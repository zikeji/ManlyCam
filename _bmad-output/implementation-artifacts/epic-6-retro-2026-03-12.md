## What Went Well ✅

### 1. Successful MediaMTX Architectural Pivot
Extracting MediaMTX from a server-managed subprocess to a standalone Docker service (Story 6-1) was a major win. It decoupled the video pipeline from the Hono application logic, simplified the `streamService.ts` code, and aligned our local development environment more closely with production.

### 2. Hardware-Verified Provisioning Automation
The `install.sh` and `uninstall.sh` scripts (Story 6-2) provide a "one-click" setup experience for the Pi. The fact that these were physically smoke-tested on a Pi Zero W 2 by the project lead ensures that our release engineering isn't just theoretical.

### 3. Monolithic Build & Runtime Env Injection
The unplanned `web-runtime-env-monolithic-deploy` story solved two major production pain points:
- It eliminated the need for a separate `manlycam-web` image.
- It solved the Vite "build-time vs runtime" environment variable problem by injecting `window.__env__` at startup.

### 4. Professional Documentation Baseline
We now have a complete set of documentation (`pi/README.md`, `docs/deploy/README.md`, and the root `README.md`). The root README with its Mermaid diagram provides an immediate, high-level understanding of the system's complex frp/MediaMTX/WebRTC pipeline.

### 5. Dynamic Code Coverage Tracking
The implementation of the dynamic coverage badge in Story 6-4 adds a layer of transparency and gamification to our quality standards, encouraging the team to maintain the high testing floor established in previous epics.

## Challenges & Learnings ⚠️

### 1. Late-Stage Environment Variable Realization
**What happened:** We realized late that Vite's build-time environment variable substitution (e.g., `VITE_PET_NAME`) didn't work for a generic Docker image meant to be configured by different operators.
**Lesson:** For any SPA that will be distributed as a containerized product, the "Runtime Env Injection" pattern should be established in Epic 1, not Epic 6. Build-time variables are only suitable for SaaS apps where the build and deploy environments are the same.

### 2. Pathing and Location Friction
**What happened:** Moving deployment configs from `apps/server/deploy/` to `docs/deploy/` mid-epic caused some friction with existing stories and documentation.
**Lesson:** The project structure for "Distribution Artifacts" (Docker Compose, systemd units) should be standardized earlier. Putting them in `docs/deploy/` is a good pattern for visibility, but the transition was slightly disruptive.

### 3. Coverage Badge Setup Dependency
**What happened:** The dynamic coverage badge requires a GitHub Gist ID and a token with Gist permissions, which the AI agent cannot create autonomously.
**Lesson:** Be more explicit in Story ACs about "External Setup" requirements that require human intervention. We have a "Technical Debt" item to finalize the Gist ID.
