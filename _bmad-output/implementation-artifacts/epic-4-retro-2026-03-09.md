## What Went Well ✅

### 1. Epic 3 Infrastructure Composing Cleanly

The WebSocket hub (`Map<string, Connection>` in 4-6), `useWebSocket.ts` composable, module-level singleton exports, and `requireAuth` middleware all composed into Epic 4 stories without architectural friction. The `onmessage` dispatch pattern from 3.4 extended to `chat:message`, `chat:edit`, `chat:delete`, `presence:*`, and `typing:*` types with zero refactoring.

The 4-6 `WsHub` upgrade to `Map<string, Connection>` for per-connection user tracking was designed for presence — and will be directly useful for Epic 5's ban-with-immediate-session-revocation story.

---

### 2. Test Pattern Maturity

Testing conventions established in Epic 3 were applied consistently across all 7 stories:
- `afterEach(() => { wrapper?.unmount() })` cleanup — caught when missing in 4-1 code review and fixed
- Module-level singleton exports for test reset capability (pattern matured through 4-1, 4-2, standardized by 4-5/4-6)
- UTC ISO strings for timezone-safe tests (4-2)
- `vi.resetModules()` isolation for composables

The result: 527 tests, all passing, TypeScript clean. The testing floor is solid going into Epic 5.

---

### 3. Adversarial Code Review as a Safety Net

Code review caught a CRITICAL bug that unit tests missed entirely: in 4-4b, Reka-UI's `Switch` fires `@checked-change`, not `@update:checked`. Unit tests mocked the event handler — they passed while the feature was silently broken. Adversarial review asked "does this PATCH request actually fire?" and caught it.

This validates the adversarial review practice. Unit test coverage and adversarial review are not redundant — they catch different failure modes.

---

### 4. Mid-Sprint Pivots Were Improvements, Not Corrections

Every UX pivot in Epic 4 produced a better outcome than the original spec:
- 4-6 typing heartbeat pattern (immediate fire + 4s keep-alive) is more reliable than the specced 400ms debounce
- 4-1 scroll state preservation across tab switches is better UX than the AC required
- 4-4b touch overlay with 3-second auto-hide is better than nothing in the spec

The pivots weren't failures — they were the implementation process discovering better solutions.

### 5. Unplanned Work Handled With Discipline

4-4b was a full unplanned story: formal story file, complete ACs, adversarial code review, sprint-status update. A CRITICAL bug in production camera controls got the same rigor as a planned feature. No silent fixes, no missing documentation.

## Challenges & Learnings ⚠️

### 1. Agent Overconfidence on Multi-Attempt Bug Fixes

**What happened (4-4b):** The switch event name bug required multiple dev agents before it was actually fixed. Each agent confidently declared "fixed" after running unit tests — which passed because the tests mocked the event and never exercised the real Reka-UI component behavior.

**Root cause:** The verification method (unit tests with mocked events) didn't match the failure mode (wrong event name on a real component). The agent was testing the wrong thing and calling it done.

**Impact:** Frustrating experience of repeated false "fixed" declarations. Erodes trust in agent output when it matters.

**Lesson:** For UI interaction bugs, "tests pass" is insufficient verification. The agent should explicitly request a smoke test of the specific interaction before declaring done — putting the final call in Zikeji's hands.

---

### 2. UX Interaction Gaps — Accepted, Not Solved

**Pattern across the epic:** Touch overlay (4-4b), typing indicator timing (4-6), scroll preservation (4-1) — all were interaction-level gaps not caught in the UX spec. The Epic 3 action item ("write interaction flows") scoped too narrowly to hover states and didn't cover all input methods (touch, keystroke timing, tab switching).

**Conscious decision:** These gaps are expected and accepted through MVP. Mid-sprint pivots via code review are the intended catch mechanism. Over-engineering the spec process for an MVP of a solo-developer dog camera project is not the right tradeoff.

**Scheduled resolution:** Full UX/interaction review after Epic 6 — once the MVP is complete.

**No action item required** — this is a deliberate product decision.
