## What Went Well ✅

### 1. **OAuth Flow — Zero Security Regressions**

**Implementation Quality:**
- Google OAuth flow implemented using native `fetch()` (no external auth libraries)
- CSRF protection: state token generated with `randomBytes(16)`, validated on callback
- OAuth state cookie uses `SameSite=Lax` (allows redirect) + `httpOnly` + 10-min expiry
- Session cookie uses `SameSite=Strict` (no cross-site leakage) + `httpOnly` + 30-day expiry
- Token exchange happens server-side (client never sees client secret)
- All sensitive data never exposed to JavaScript

**Code Review Findings:**
- Story 2.1 review caught: SPA catch-all path used relative paths (broken in production) → fixed with absolute paths via `import.meta.url`
- Story 2.1 review caught: logout error handling was missing → added response status check before clearing client state
- Story 2.1c code review caught: OAuth callback route test was fake-passing (mocked to return `undefined`) → added proper happy-path test

**Why This Matters:**
Using native `fetch()` made OAuth implementation transparent and reviewable. No library abstractions meant developers had to understand every step, making security issues visible during code review.

**Result:** Production-ready OAuth with zero rework cycles.

### 2. **Allowlist Logic — Clean Role Mapping**

**Design Decision:**
- New users matched against allowlist in order: domain first, then email, else reject
- Domain match → `role: 'ViewerCompany'`
- Email match → `role: 'ViewerGuest'`
- No match → no DB row created, no session, redirect to `/rejected`
- Returning users: skip allowlist (already trusted), only check `bannedAt`

**Type Safety:**
Return type changed from `Promise<{ sessionId: string }>` to `Promise<{ sessionId: string | null }>`. This **forces** the route handler to ask "do I have a session?" before setting the cookie. Catches potential bugs at compile time.

**Code Quality:**
- Single source of truth: role is implicit in allowlist entry type
- Fail-secure: default is reject; only explicit allowlist entries grant access
- Tested: domain match, email match, no-match, and ban-check paths all have specific test cases

**Result:** Allowlist enforcement that's both strict and maintainable.

## Challenges & Learnings ⚠️

### 1. **SPA Catch-All Path — Relative vs. Absolute**

**The Issue:**
Story 2.1 added production SPA serving via:
```typescript
// BROKEN — relative path fails if working directory changes
app.use('/*', serveStatic({ root: '../web/dist' }))
```

**Root Cause:**
File paths are relative to the current working directory. In production, the working directory might not be what the developer expects. This is environment-specific and hard to debug.

**The Fix (Code Review):**
Use absolute path via `import.meta.url`:
```typescript
import { fileURLToPath } from 'node:url'
import { dirname } from 'node:path'

const __dirname = dirname(fileURLToPath(import.meta.url))
const distPath = resolve(__dirname, '../web/dist')
app.use('/*', serveStatic({ root: distPath }))
```

**Lesson:**
File paths in route handlers must be absolute or relative to a fixed reference point. This is not Node.js specific — it's a production deployment gotcha. Always use `import.meta.url` for module-relative paths.

**Applied to Future Stories:**
All file operations in routes now use absolute paths computed from `import.meta.url`.

### 2. **OAuth State Cookie — SameSite Flag Nuance**

**The Issue:**
OAuth state must survive a cross-origin redirect:
1. User clicks "Sign in" on `example.com`
2. Redirects to `accounts.google.com`
3. Google redirects back to `example.com/callback`
4. State cookie must survive step 3

**Why It Matters:**
- `SameSite=Strict` drops cookies on all cross-site requests → state validation fails
- `SameSite=Lax` allows top-level navigation but blocks subresources → correct for OAuth
- Session cookie is `SameSite=Strict` (no cross-site leakage)
- OAuth state cookie is `SameSite=Lax` (allows OAuth redirect)

**Code Discovery (Story 2.1c):**
Code review added **explicit tests** verifying the OAuth state cookie has `SameSite=Lax`. If someone accidentally changed it to `Strict` in a future refactor, CI would fail.

**Lesson:**
Security-critical details shouldn't be documented and forgotten — they should be test-enforced. CI catches accidental regressions.

**Applied to Future Stories:**
All security-sensitive cookie operations now have explicit test assertions for `SameSite`, `httpOnly`, `Secure`, and expiry flags.

### 3. **Gravatar Fallback — Email Normalization**

**The Issue (Story 2.4):**
Avatar resolution needed a fallback when Google doesn't provide a picture. Gravatar is computed via MD5 hash of the email address.

**The Gotcha:**
```typescript
// WRONG — case-sensitive hash
const hash = createHash('md5').update(email).digest('hex');
// user@example.com → one hash
// User@Example.com → different hash
// Gravatar won't find the avatar

// RIGHT — normalize first
const hash = createHash('md5').update(email.toLowerCase().trim()).digest('hex');
// Both resolve to same hash
```

**Why It Matters:**
Gravatar's hash is case-insensitive, but only if the input is normalized. Missing this means different email cases get different avatars (or no avatar).

**Code Quality:**
The implementation correctly normalizes. The dev notes explicitly called out the edge case, making it visible to reviewers.

**Lesson:**
Cryptographic operations (hashing, encoding) have subtle rules beyond "just call the function." Document assumptions; test edge cases.

**Applied to Future Stories:**
Any cryptographic operations now have explicit documentation of normalization rules and test cases for edge cases.

### 4. **Allowlist Upsert vs. Insert — Idempotency Trade-off**

**The Decision (Story 2.5):**
When adding a domain to the allowlist twice, should we:

**Option A: Upsert (Idempotent)**
```typescript
await prisma.allowlistEntry.upsert({
  where: { type_value: { type, value } },
  create: { id: ulid(), type, value },
  update: {},  // no-op
});
// Running twice = silent success both times
```

**Option B: Insert + Error Handling**
```typescript
try {
  await prisma.allowlistEntry.create({...});
} catch (err) {
  if (err.code === 'P2002') throw new Error('Already exists');
}
// Running twice = error on second run
```

**Decision Rationale:**
- Operators might re-run commands out of habit or safety
- Idempotent operations (upsert) are safer than error-handling
- Simpler code, fewer edge cases
- Matches Unix philosophy: running `add-domain company.com` twice should be safe

**Lesson:**
When a unique constraint exists and you want safe, operator-friendly CLI, `upsert` is cleaner than error handling. Idempotency matters for tools that humans operate.

### 5. **Ban Atomicity — Avoiding Partial Failures**

**The Requirement (Story 2.5, AC #5):**
When banning a user, BOTH operations must succeed or BOTH must roll back:
1. Set `users.bannedAt = NOW()`
2. Delete all `sessions` rows for that user

**Why It Matters:**
- If we update `bannedAt` but crash before deleting sessions → user is "banned" but has active sessions (contradiction)
- If we delete sessions but crash before updating `bannedAt` → sessions are gone but `bannedAt` is null, so user can log back in (security hole)
- The operation must be **all or nothing**

**Solution:**
```typescript
const [, deletedSessions] = await prisma.$transaction([
  prisma.user.update({
    where: { id: user.id },
    data: { bannedAt: new Date() },
  }),
  prisma.session.deleteMany({ where: { userId: user.id } }),
]);
return { sessionCount: deletedSessions.count };
```

**Code Review Note:**
A TODO was added for Story 3.4: `// TODO(story-3.4): WS hub will detect missing session and emit session:revoked`. The ban sets up the precondition (no sessions), but the real-time signal happens in the next epic.

**Lesson:**
Atomic transactions prevent partial-failure states that are hard to debug. Use `prisma.$transaction()` for multi-step operations where consistency matters.
